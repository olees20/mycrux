# 015 — Interactive gym floorplan

## Decision

The gym's visual navigation starts with one `gym_floorplans` document per gym and metre-based `wall_structures`. A structure is a physical, top-down wall segment; the existing `walls` records remain climbing faces/areas used by routes. Their nullable `wall_structure_id` is deliberately only an attachment boundary for a later phase.

The browser edits a complete floorplan document. `save_gym_floorplan` validates and writes it in one database transaction, derives authorization from the authenticated session, archives removed structures, records one audit event, and increments an optimistic revision. A stale editor cannot silently overwrite a newer save.

## Geometry and scale

- Coordinates and thickness use metres with millimetre persistence precision.
- `length_metres` is generated by PostgreSQL from the endpoints, preventing client/database disagreement.
- The current editor supports up to 5,000 wall structures per gym; indexed active-floorplan reads are linear in the returned document and comfortably cover commercial sites with hundreds of walls.
- UUIDs make disconnected and adjoining structures equivalent in storage. Adjoining is a geometric relationship created by endpoint snapping, not a fragile parent/child chain.
- Removed walls are archived rather than destroyed, preserving the path to history, maintenance, analytics, and future face attachments.

## Authorization

Active gym members may read the map, allowing it to become member navigation later. Only owners can provision or save it. Authenticated users have no direct insert/update/delete grants on the new tables; all mutations pass through the owner-only RPC with RLS remaining enabled.

## Deferred deliberately

Wall faces, holds, volumes, routes, AR anchors, inventory, and competition overlays are outside Phase 1. No placeholder hold or route model is introduced here.
